# План доработки пользовательского интерфейса (Vision Board)

На основе документа **Vision Board.docx**: список фич, скриншоты и структура. Цель — привести дизайн и функциональность страниц пользователя (User / GSOC) в соответствие с видением.

### UI-примеры из документа (со скриншотами)

- **Response by the user to the incident** — как может выглядеть ответ пользователя на инцидент от security team (без эмодзи).
- **Reporting a new incident** — как может выглядеть форма/страница создания нового инцидента.
- **Incidents tab** — вкладка инцидентов с фильтрами для поиска.
- **Time range filter** — при поиске инцидентов пользователем: «Presents + Date & Time Range».

Скриншоты из docx извлечены в `vision-board-extract/word/media/` (image1.png … image21.jpg) и могут использоваться как референсы при верстке.

## Текущее состояние приложения

- **Роли:** End-User, Security Manager, GSOC, GRC, IAM, Pentesting, Admin.
- **Сайдбар (SideMenu):**
  - End-User: Requests, Report Activity.
  - Premium (не End-User): Requests, Reported Activity.
  - Admin: ссылка на Admin.
- **Страницы:**
  - `/` — логин/регистрация.
  - `/tickets` — заявки (Requests).
  - `/report` — репорты/инциденты (Report Activity / Reported Activity).
  - `/admin/*` — админка (users, requests, auth).

В приложении уже есть: тёмная тема (Dark mode), базовая структура сайдбара с логотипом и ролью, разделение по ролям.

## План реализации (по приоритету)

### Фаза 1 — Профиль и базовая навигация (дизайн страницы юзера)

1. **User profile (фича 1)**  
   - Отдельная страница/раздел «Профиль» (или модальное окно из сайдбара).  
   - Поля: аватар, email, имя/фамилия (уже есть), **job title**, **department**.  
   - Бэкенд: расширить модель пользователя (job_title, department), API для GET/PATCH профиля.  
   - В сайдбаре показывать аватар и имя (или «Имя Ф.») по текущим данным.

2. **Settings (фича 5)**  
   - Страница «Настройки»: настройки приложения (язык, тема) и аккаунта (профиль, смена пароля и т.д.).  
   - Добавить пункт в сайдбар (иконка шестерёнки) или объединить с профилем (вкладки Профиль / Настройки).

3. **Language settings (фича 2)**  
   - В настройках или в хедере: переключатель языка (кнопка «планета» или выпадающий список).  
   - i18n: подготовить структуру локалей (en/ru и т.д.) и переключатель.

### Фаза 2 — Инциденты (Report Activity) и поиск

4. **Incidents list (фича 3)**  
   - Привести список инцидентов/репортов к виду с Vision Board: карточки/таблица, кнопка «Create new Incident» сверху.  
   - Единый вид для User и GSOC с разницей только в правах (создание, просмотр своих vs всех).

5. **Reporting a new incident**  
   - Отдельная страница или модальное окно создания инцидента по макету из документа (скрин «Reporting a new incident»).

6. **Incidents tab + фильтры**  
   - Вкладка инцидентов с фильтрами, как на скриншоте: поиск, статус, приоритет, дата и т.д.  
   - **Time range filter (Presents + Date & Time Range)** — реализовать компонент выбора периода (Presets + произвольный диапазон дат/времени).

7. **Response to incident**  
   - Экран/форма ответа пользователя на инцидент от security team — по скриншоту «Response by the user» (без эмодзи, структурированные поля).

8. **Search incident DB (фича 13)**  
   - На странице создания инцидента: блок «Search incident DB» — поиск по похожим прошлым инцидентам (по заголовку/описанию/тегам).

### Фаза 3 — Дополнительные разделы и UX

9. **Search bar (фича 8)**  
   - Глобальная строка поиска в хедере/сайдбаре (поиск по инцидентам, тикетам, при необходимости по пользователям).

10. **Dashboard analytics (фича 9)**  
    - Новая страница «Dashboard» с аналитикой (графики, метрики по инцидентам/тикетам).  
    - Пункт в меню для ролей с доступом.

11. **Timeline under user's incidents (фича 12)**  
    - В представлении User: таймлайн инцидентов (созданные/участие в ответах) — вертикальный таймлайн по датам.

12. **User guide (фича 4)**  
    - Раздел «User guide»: загрузка/отображение документа (PDF или статичная страница) об использовании системы.

13. **Help / Support (фича 10)**  
    - Страница или ссылка: контакты поддержки, FAQ, форма обращения.

14. **Notifications (фича 6)**  
    - UI уведомлений (иконка колокольчика, выпадающий список или отдельная страница).  
    - Бэкенд: модель уведомлений и API — отдельная подзадача.

15. **Security on-call (фича 7)**  
    - Вкладка/пункт меню «Security on-call» для User: кнопка вызова security вне рабочих часов (звонок/ссылка).  
    - Позже: расписания дежурств по командам (IAM, GSOC, Pentesting).

### Фаза 4 — Интеграции и продвинутые фичи

16. **Integrations (фича 14)**  
    - Раздел «Integrations»: SIEM (Splunk, Azure Sentinel, Elastic, Google Chronicle, Rapid7 и т.д.), создание тикетов из звонков и писем.  
    - Требует проектирования API и конфигурации интеграций.

## Рекомендуемый порядок работ для «дизайна страницы юзера»

Если фокус именно на **дизайне страницы пользователя** (внешний вид и структура):

1. Ввести **профиль пользователя** (аватар, job title, department) и вынести в отдельную страницу или блок в сайдбаре.  
2. Добавить **Settings** (и при необходимости объединить с профилем).  
3. Привести **список инцидентов** (Report Activity) к макету из Vision Board: список + кнопка «Create new Incident», карточки/таблица.  
4. Реализовать **форму создания инцидента** и **форму ответа на инцидент** по скриншотам.  
5. Добавить **фильтры** на вкладке инцидентов (в т.ч. Time range: Presets + Date & Time Range).  
6. Добавить в навигацию пункты **Dashboard**, **User guide**, **Help/Support**, **Security on-call** (заглушки или минимальная реализация), **Search bar** в шапке.

После этого можно поочерёдно добавлять Language settings, Notifications, Search incident DB, Timeline, Integrations и т.д.
